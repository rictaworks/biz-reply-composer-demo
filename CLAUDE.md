# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## プロジェクト概要

- **課題:** 文脈に合わせてビジネスメール返信文を生成するAI（デモ版・ショーケース）
- **形態:** デスクトップアプリ（Rust + Tauri / React + TypeScript / SQLite / ローカルLLM `ollama`）
- **デモ制約:** 外部API・APIキー・認証を一切使わない。ネットワーク越しの呼び出しなし。端末ローカルのセッションIDのみ。
- **詳細仕様（正）:** `@biz-reply-composer-demo_instruction.md`（仕様・ER図・DFD・シーケンス図・クラス図・状態遷移図・ユースケース図）。実装判断の根拠は必ずここを参照する。

> 標準の開発ディレクティブはWebアプリ前提だが、本プロジェクトはデスクトップデモである。
> 認証・自動ログイン・Webページ・REST API・クラウドデプロイ（Vercel/Railway/Google認証）は**本エディションには存在しない**。
> 「API一覧」＝Tauriコマンド一覧、「ページ一覧」＝アプリ画面一覧、「デプロイ」＝ビルド と読み替える。

## Claude Safety Rules

### 削除系コマンドの禁止（重要）

以下のルールはこのワークスペース内のすべての会話で絶対に守られる：

- Claude はファイルまたはディレクトリを削除するコマンドを一切生成してはならない。
  例：rm, rm -rf, rm *, rmdir, unlink, cache --delete,
      lftp mirror --delete, rsync --delete, git clean -df, find -delete 等。
- 削除が必要な場合でも、Claude は削除コマンドを提案せず、「手動で削除してください」といった説明に留めること。
- 削除の推奨・削除操作の自動判断も禁止。不要物は `DELETE/`（ゴミ箱）へ `mv` する。
- ssh / lftp / デプロイ系スクリプトを生成する場合でも、削除コマンドの生成は禁止。

これらはすべての会話・コード生成に適用される。

### シークレット管理（重要）

- `config/master.key` など機密ファイルを `git add` するコードを生成してはならない。デプロイスクリプト・セットアップ手順でも同様。
- シークレットは必ず環境変数（`RAILS_MASTER_KEY` 等）で渡すこと。環境変数は `.env` を参照する。
- `.gitignore` への追加を確認する手順を必ずコードに含めること。
- 初回コミット前に `git status` でステージング確認を促すこと。

## ブランチ・PR運用

- **`main` ブランチでの作業を禁止**する。作業前に必ずブランチを切る。
- `src/*` **以外**の変更（ドキュメント・SPEC・TASKS 等）は `main` への push を許可する。
- `src/*` の変更は **PR を作成しなければならない**（`main` への直接 push 禁止）。
- **コミット前に必ず security review を行う**（`/security-review`）。あわせて `DC.md` / `QC10.md` / `TM.md` / `QA.md`（OWASP Top 10）を参照する。
- PR は**日本語**で書く。**非エンジニア向けのユーザーテスト手順**を本文に丁寧に記載する（`pr-checker` エージェント運用）。

## 開発ワークフロー — TDD厳守

順序を厳守：**plan → red test（失敗するテストを先に書く）→ coding → green test**。

- Rust: `cargo test` / フロント（React+TS）: `vitest`（または `jest`）。RSpec 等の慣習に準拠。
- フロントの動作確認は `curl` / `wget --mirror` / `playwright` で行う。
- **フォールバック禁止。** ollama 未起動・モデル未導入・タイムアウトはすべて明示的なエラーとし、ルールベース等の代替生成に切り替えない（仕様 §1.5/§8 準拠）。
- **例外処理を厳格に書く。** 握り潰し禁止。デバッグトレースできるよう、各処理段階でログ（phase/result/elapsed_ms）を残す。

## コーディング規約

- **グローバル変数を禁止**する（セキュリティ観点）。制御構文・条件構文以外のロジックは、必ずクラスまたは関数に閉じる。
- **文字列リテラルは設定ファイルまたはDBに分離**する。ハードコードを検出するテストを必ず書く。
- **デフォルトアイコンは Font Awesome** を使用する。**絵文字は禁止。**
- **ネイティブ `alert()` / `confirm()` / `prompt()` はプロジェクト全体で使用禁止。**
- **環境判定を必ず実装し分岐可能にする。** テスト容易性のため、開発環境ではセッションを自動初期化するなど、確認ステップを短絡させてよい（本デモは認証なしのため「認証済み分岐」は該当なし）。
- 車輪の再発明を避け、安全なライブラリ・OSS を優先。ただしデモ制約（外部API・ネットワーク禁止）を犯すSaaSは採用しない。

## 多言語（i18n）

- 当初から多言語で開発する：**日本語・英語・フランス語・中国語・ロシア語・スペイン語・アラビア語**。
- **開発者用の管理画面は日本語のみ**。
- 返信文はユーザーの入力言語に合わせて生成する（仕様 §1.5 手順3）。

## サブエージェント（規模に応じて起用）

`director` / `project-manager` / `designer` / `debugger` / `tester` / `data-scientist` / `deployer` / `writer` / `service-manager` / `pr-checker`。定義は `.claude/agents/` を参照。

- `pr-checker`: レビューはしない。全PRを日本語化し、非エンジニア向けユーザーテストを本文に丁寧に書く。
- `tester`: 全PR対象。PR本文のユーザーテスト手順の実行スクリプトと `TM.md` 記載のテストを作成。テストは `test/pr***/` に置き、対象は**開発サーバー**とする。

# ユーザーテスト手順（biz-reply-composer-demo）

対象：[PR #2](https://github.com/rictaworks/biz-reply-composer-demo/pull/2)（既定モデルを `gemma3:1b` に変更・欠落フィールド許容）マージ後の実機確認。

Tauriデスクトップアプリのため、実際の起動・クリック確認はユーザーが行う（Claudeはcomputer useでの操作は行わない）。

## 前提

- ollamaが起動しており、`gemma3:1b` が導入済みであること（`ollama list` で確認）。
- 未導入なら `ollama pull gemma3:1b`。

## 起動

```powershell
cd E:\Github\rictaworks\biz-reply-composer-demo
npx tauri dev
```

## 0. 既定モデルの確認

- 設定でモデルを明示していない状態で起動し、生成に使われるモデルが `gemma3:1b` であること（ログ or 応答速度感）を確認する。
- コールドロード込みで1回目の生成が概ね20〜30秒程度で完了すること（タイムアウト30秒に収まること）を確認する。

## 1. ゴールデンパス

1. 受信メール本文欄に、依頼・催促・謝罪・日程調整・クレームのいずれかの例文を貼り付ける（10〜8,000字）。
2. 返信方針（承諾／辞退／保留／追加質問）を1つ選ぶ。
3. トーン（フォーマル／標準／カジュアル）を1つ選ぶ。
4. 生成ボタンを押し、以下を確認する。
   - 文脈解析結果（用件カテゴリ・要求事項・期日・相手の感情トーン）が表示される。
   - 返信ドラフトが宛名→挨拶→本文→結びの4部構成で表示される。
   - 宛名・署名が `{{相手名}}` `{{自分の署名}}` トークンのままで、実名が捏造されていない。
5. ワンタップ再生成（「もっと丁寧に」等）を1つ試し、再生成結果が反映されることを確認する。
6. 履歴表示で、直前の生成が一覧に出てコピーできることを確認する。

## 2. 異常系（§1.6 に基づく）

- 空入力で生成 → エラー表示（フォールバックしない）
- 10文字未満で生成 → エラー表示
- 8,000字超の本文 → 要点圧縮後に生成が完了する
- 英語メール本文 → 返信も英語で生成される
- ollamaを停止した状態で生成 → 明示的なエラー（フォールバックしない）
- 未導入モデルを指定して生成 → 明示的なエラー
- 氏名・メールアドレス・電話番号を含む本文 → 保存データが伏字化されている（DB直接確認、またはアプリ上で個人情報が露出していないこと）

## 3. 今回の変更に関する確認（PR #2 固有）

- LLMの文脈抽出結果に `senderSentiment` や `requests` が欠けたレスポンスが返っても、アプリがクラッシュ・解析エラーにならず処理が継続すること（意図的に再現するのは難しいため、複数回生成して解析エラーが出ないことで代替確認可）。
